  <action>
Create new file. Istio AuthorizationPolicy ALLOW rule restricting portfolio-api ingress to two principals only (D-08). Per Istio semantics, when any ALLOW policy matches a workload, traffic from anything NOT matching is DENIED by default — so this is a hard lock-down, not just a preference.

```yaml
# D-08 — Reinforce the NetworkPolicy with mesh-level mTLS identity checks.
# Only the portfolio-web workload and the Istio ingress gateway may call portfolio-api.
apiVersion: security.istio.io/v1
kind: AuthorizationPolicy
metadata:
  name: portfolio-api-allow
  namespace: portfolio
spec:
  selector:
    matchLabels:
      app: portfolio-api
  action: ALLOW
  rules:
    - from:
        - source:
            principals:
              - "cluster.local/ns/portfolio/sa/portfolio-web"
              - "cluster.local/ns/istio-ingress/sa/istio-ingressgateway-service-account"
```

Rationale for not putting this inside the chart: base/ files target the cluster operator's concerns (mesh policy, ingress wiring). The chart stays environment-agnostic; cluster-specific identity strings (like the ingress gateway's SA name) live here. This keeps the chart reusable if someone ever tests it in a different cluster.

Note on the ingress gateway SA name: the upstream Istio Helm chart default is `istio-ingressgateway-service-account` in namespace `istio-ingress`. If the local install used a different name, this string must be adjusted. Plan 03-06's human checkpoint (or the Phase 4 cutover verification) is the place to confirm.
