<task type="checkpoint:decision" gate="blocking">
  <name>Task 2: Decide HelmRelease semver pin strategy</name>
  <decision>Whether `portfolio/base/helmrelease.yaml` should pin `spec.chart.spec.version` or track latest.</decision>
  <context>
Plan 03-03 intentionally left the HelmRelease unpinned so Flux uses the latest published chart version. Now that the chart bumps to 0.2.0, we have two reasonable policies:
  </context>
  <options>
    <option id="option-a">
      <name>Track latest (no pin)</name>
      <pros>Chart pushes roll out automatically — simpler, matches the image-automation philosophy.</pros>
      <cons>A bad chart push instantly affects prod; no explicit "rollout of chart X" commit in git.</cons>
    </option>
    <option id="option-b">
      <name>Pin semver range (`">=0.2.0 &lt;0.3.0"`)</name>
      <pros>Minor bumps require an explicit HelmRelease commit — a visible, reviewable rollout gate.</pros>
      <cons>Slightly more ceremony; drift between chart version and HelmRelease if forgotten.</cons>
    </option>
    <option id="option-c">
      <name>Pin exact (`"0.2.0"`)</name>
      <pros>Strictest — every chart bump requires a HelmRelease edit; easiest rollback.</pros>
      <cons>Defeats the purpose of automated chart publishing for patch fixes.</cons>
    </option>
  </options>
  <resume-signal>Select: option-a, option-b, or option-c. Claude will then edit portfolio/base/helmrelease.yaml accordingly.</resume-signal>
</task>
