Per D-08: Mesh-wide STRICT mode requires per-namespace PERMISSIVE overrides where needed. Per RESEARCH.md Pitfall 5 and Open Question 3, the monitoring namespace may not have Istio sidecar injection enabled. If Prometheus lacks a sidecar, STRICT mTLS will break metrics scraping (Prometheus cannot present a valid mTLS certificate). A PERMISSIVE override allows both plaintext and mTLS connections to monitoring pods, ensuring scraping works regardless of sidecar status.

**Create `platform-tools/eks-monitoring/base/peerauthentication.yaml`:**

```yaml
# PERMISSIVE mTLS override for monitoring namespace.
# Allows both plaintext and mTLS connections.
# Required because Prometheus may not have an Istio sidecar,
# and STRICT mode would break metrics scraping.
# See RESEARCH.md Pitfall 5: "Monitoring Scraping Breaks with STRICT mTLS"
apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata:
  name: default
  namespace: monitoring
spec:
  mtls:
    mode: PERMISSIVE
```

**Update `platform-tools/eks-monitoring/base/kustomization.yaml`** -- add `peerauthentication.yaml` to the resources list after `grafana-sealed-secret.yaml`:

```yaml
  - helmrepository.yaml
  - helmrelease.yaml
  - grafana-sealed-secret.yaml
  - peerauthentication.yaml
```

This override is namespace-scoped (metadata.namespace: monitoring) and only affects pods in the monitoring namespace. All other namespaces inherit the mesh-wide STRICT policy from istio-system.
